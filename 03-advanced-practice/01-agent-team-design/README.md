# 01. 에이전트팀 설계

> 한 명의 AI로 안 되는 일을 팀으로 처리한다

---

## 왜 에이전트팀인가?

혼자 하는 AI의 한계:
- 한 번에 처리할 수 있는 정보량 제한
- 모든 것을 순차적으로 처리 → 느림
- 하나의 관점만 반영

에이전트팀의 해결:
- 작업을 나눠서 병렬 처리 → 빠름
- 각 역할에 특화된 AI → 품질 향상
- 서로 검토하고 보완 → 안정성

---

## 에이전트팀의 기본 구조

```
[사용자]
    ↓ 목표 전달
[Orchestrator (총괄)]
  - 목표를 이해하고 계획 수립
  - 작업을 Worker들에게 분배
  - 결과를 취합하고 검토
    ↓ ↓ ↓ (동시 실행)
[Worker A]  [Worker B]  [Worker C]
전문 분야A  전문 분야B  전문 분야C
    ↓ ↓ ↓ (결과 반환)
[Orchestrator]
    ↓ 통합 결과
[사용자]
```

---

## 역할 설계 예시

### 예시 1: 마케팅 콘텐츠 팀

```
[Orchestrator] 콘텐츠 전략 담당
    ├── [Research Agent] 트렌드·경쟁사 조사
    ├── [Writer Agent] 텍스트 콘텐츠 작성
    ├── [SEO Agent] SEO 최적화 검토
    └── [Review Agent] 브랜드 톤 일관성 검토
```

하나의 블로그 포스트를 만들 때:
- Research: 주제 관련 최신 동향 수집
- Writer: 초안 작성
- SEO: 키워드 최적화
- Review: 브랜드 가이드라인 준수 확인
- Orchestrator: 최종 통합 및 교정

---

### 예시 2: 고객 서비스 팀

```
[Orchestrator] 라우터 역할
    ├── [FAQ Agent] 자주 묻는 질문 처리
    ├── [Complaint Agent] 불만 처리 특화
    ├── [Technical Agent] 기술 문의 처리
    └── [Escalation Agent] 긴급 건 판단 및 에스컬레이션
```

---

### 예시 3: 리서치 분석 팀

```
[Orchestrator] 분석 방향 설정
    ├── [Data Collector] 자료 수집 (웹, 파일)
    ├── [Quantitative Agent] 정량 데이터 분석
    ├── [Qualitative Agent] 정성 데이터 분석
    └── [Report Writer] 최종 보고서 작성
```

## 설계 원칙

### 1. 역할을 명확하게

```
나쁜 역할:   "뭐든지 하는 Agent"
좋은 역할:   "경쟁사 가격 정보만 수집하는 Agent"
```

### 2. 작업을 독립적으로 분리

```
의존성 있음 (병렬 불가):
  수집 → 분석 → 보고서  (순서 중요)

독립적 (병렬 가능):
  A사 분석 + B사 분석 + C사 분석 동시에
```

### 3. 결과 형식을 통일

각 Worker가 같은 형식으로 결과를 반환해야 Orchestrator가 쉽게 통합합니다.

```
나쁜 예: Worker A는 JSON, Worker B는 마크다운, Worker C는 자유 텍스트
좋은 예: 모두 같은 JSON 구조로 반환
```

## 에이전트팀 설계 워크시트

내 업무에 에이전트팀을 적용해보세요:

**1단계: 목표 정의**
```
이 팀이 처리할 반복 업무는?
최종 결과물은?
```

**2단계: 단계 분해**
```
이 업무를 순서대로 나열하면?
각 단계가 독립적인가, 순차적인가?
```

**3단계: 역할 정의**
```
필요한 전문성은?
각 Agent의 이름과 역할은?
```

**4단계: 인터페이스 정의**
```
각 Agent의 입력은?
각 Agent의 출력 형식은?
```
