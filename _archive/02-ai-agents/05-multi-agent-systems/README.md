# 05. 멀티 에이전트 시스템

---

## 1. 멀티 에이전트 시스템이란?

여러 Agent가 협력하여 단일 Agent로는 불가능한 복잡한 목표를 달성하는 시스템.

```
단일 Agent 한계:
  - 컨텍스트 윈도우 한계
  - 동시 처리 불가
  - 단일 전문성

멀티 Agent 해결:
  - 작업 분산
  - 병렬 처리
  - 전문화된 역할
```

---

## 2. 실제 사례: 소프트웨어 개발 팀 시뮬레이션

```
[PM Agent]
  → 요구사항 분석, 태스크 분해

[Architect Agent]
  → 기술 스택 결정, 시스템 설계

[Frontend Agent]          [Backend Agent]
  → UI 구현                 → API 구현

[QA Agent]
  → 테스트, 버그 리포트

[DevOps Agent]
  → 배포, 인프라
```

---

## 3. Claude Code 멀티 에이전트 실습

### 기본 멀티 에이전트 호출

```bash
claude
```

```
다음 작업을 멀티 에이전트로 처리해줘:

목표: 이 Node.js 프로젝트의 종합 감사

에이전트 구성:
1. [Security Agent]: 보안 취약점 스캔
2. [Performance Agent]: 성능 병목 분석
3. [Quality Agent]: 코드 품질 및 베스트 프랙티스
4. [Dependency Agent]: 패키지 의존성 및 업데이트 필요 항목

각 에이전트 결과를 종합해서 우선순위 있는 액션 플랜을 줘.
```

---

## 4. 실전 프로젝트: 자동화 파이프라인

**목표**: 매일 아침 자동으로 팀에 브리핑 생성

```
[Trigger: 매일 09:00]
      ↓
[Collector Agent]
  - GitHub: 어제 PR, 이슈
  - Jira: 진행 중 태스크
  - Slack: 주요 대화 요약
      ↓
[Analyzer Agent]
  - 블로커 감지
  - 진행률 계산
  - 리스크 평가
      ↓
[Writer Agent]
  - 브리핑 문서 작성
  - 핵심 알림 생성
      ↓
[Distributor Agent]
  - Slack 발송
  - 이메일 발송
  - Notion 업데이트
```

---

## 5. 멀티 에이전트 설계 원칙

### DO (해야 할 것)

```
✓ 명확한 역할 분리
✓ 표준화된 인터페이스 (Agent 간 통신 형식 통일)
✓ 실패 처리 (하나 실패 시 전체 영향 최소화)
✓ 로깅 (각 Agent의 결정 기록)
✓ 사람이 개입할 포인트 설계
```

### DON'T (하지 말 것)

```
✗ Agent가 너무 많음 (관리 복잡도 폭발)
✗ Agent 간 순환 의존성
✗ 사람 검토 없이 중요 결정 자동화
✗ Agent에게 너무 많은 권한 부여
```

---

## 6. 비용과 트레이드오프

| 요소 | 단일 Agent | 멀티 Agent |
|------|-----------|-----------|
| 속도 | 느림 (순차) | 빠름 (병렬) |
| 비용 | 낮음 | 높음 |
| 복잡도 | 낮음 | 높음 |
| 품질 | 중간 | 높음 (전문화) |
| 디버깅 | 쉬움 | 어려움 |

**언제 멀티 에이전트가 정당화되나?**
- 작업 시간 단축이 비용보다 가치 있을 때
- 단일 Agent의 품질이 충분하지 않을 때
- 정기적으로 반복되는 복잡한 작업일 때

---

## 7. 시작하기 위한 첫 걸음

**처음이라면 이렇게 시작하세요:**

1. **단순하게**: 먼저 단일 Agent로 문제를 해결
2. **병목 찾기**: 어디서 시간이 오래 걸리나?
3. **하나만 분리**: 가장 독립적인 작업 하나를 Subagent로
4. **점진적 확장**: 효과를 확인하며 Agent 추가
